## Approach

The verification strategy follows the architecture split between the new contract-definition path, the unchanged constructor-resolution path, and the repository-topology move [ref: ./01-architecture.md#module-boundaries] [ref: ./05-usecases.md#consumer-use-cases].

- Unit testing covers contract-object identity, normalization, missing-binding behavior, rebinding state rules, scoped compatibility semantics, and constructor-path non-regression inside the core runtime [ref: ./01-architecture.md#contract-definition-architecture] [ref: ./02-dataflow.md#flow-4-rebinding-before-first-resolution].
- Integration testing covers root-package consumption through `inject.define`, React scope behavior for bound contracts, coexistence of contract and constructor call sites, and export-surface guarantees from the package barrel [ref: ./01-architecture.md#system-fit] [ref: ./05-usecases.md#uc-2-resolve-the-contract-through-inject-inside-existing-application-code] [ref: ./05-usecases.md#uc-3-use-a-bound-scoped-contract-in-react-without-creating-a-separate-react-specific-api].
- Repository-level regression testing covers relocation into `src/core/__tests__` and `src/react/__tests__`, alias-based import behavior for moved React hook tests, unchanged discovery/config assumptions, and synchronized docs/changelog/export verification [ref: ./01-architecture.md#test-topology-organization] [ref: ./04-decisions.md#adr-6-standardize-relocated-react-hook-tests-on-alias-based-imports] [ref: ./07-docs.md#required-documentation-touchpoints].

## Edge Cases

- Repeated `define("SameName")` calls in different modules must be verified as distinct contracts so that docs and tests do not accidentally assume name-based identity [ref: ./03-model.md#identity-rule].
- Rebinding immediately before the first resolution should replace the pending descriptor, but a bind attempt after any first `getInstance()` path must be rejected for singleton, scoped, and transient lifetimes alike [ref: ./04-decisions.md#adr-4-allow-rebinding-only-before-first-resolution].
- Bound scoped contracts must be tested both inside and outside `DiScopeProvider` to confirm that binding changes registration semantics but does not remove scope-presence requirements [ref: ./04-decisions.md#adr-5-treat-binding-as-registration-for-scoped-contracts-while-preserving-existing-scoped-runtime-rules].
- Export verification must cover the absence of any newly required named contract export, because the architecture intentionally keeps the package API centered on `inject.define` only [ref: ./04-decisions.md#adr-1-keep-the-public-api-centered-on-injectdefine].
- Relocated test files must not create duplicate discovery by leaving old source-adjacent copies behind or by changing suffixes away from the current Vitest include contract [ref: ../01-research/02-test-topology-analysis.md#scope-boundaries-and-observed-discovery-dependencies].

## Performance Criteria

No separate performance gate is required for this design. The approved architecture adds only constant-time contract-state lookup around the existing resolver and keeps the current test-discovery globs unchanged; QA should only watch for accidental duplicate discovery or duplicated suite execution after relocation [ref: ./02-dataflow.md#data-flow-invariants] [ref: ./04-decisions.md#adr-7-keep-vitest-and-typescript-discovery-configuration-unchanged-for-the-chosen-topology].

## Correctness Verification

Correctness is established when all three verification layers agree with the approved design:

1. Unit coverage proves the contract path obeys identity, missing-binding, rebinding, scoped-compatibility, and constructor non-regression rules [ref: ./01-architecture.md#architectural-invariants].
2. Integration coverage proves a consumer can use `inject.define` from the root package surface and can resolve a bound scoped contract through the existing React API without a parallel registration mechanism [ref: ./05-usecases.md#consumer-use-cases].
3. Repository regression coverage proves the relocated unit tests are still discovered under the unchanged config contract, the React hook tests are import-stable from their new paths, and the public docs/export/changelog surfaces all describe the same behavior [ref: ./05-usecases.md#repository-maintainer-use-cases] [ref: ./07-docs.md#required-public-facing-verification-touchpoints].
